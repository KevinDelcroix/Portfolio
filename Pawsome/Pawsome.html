Gameplay

Pawsome is divided into two main gameplay phases: Living Room Phase Before and after battles, the player will have access to a HUB set in a cozy living room that serves as a habitat for the cats they‚Äôve collected. During this phase, the player can prepare for the next fight by upgrading buildings using kibble in order to generate more resources and house more cats. To activate production buildings, the player must assign one of their cats to them."

Combat Phase During combat, players enter an 8x8 arena. In this phase, each player uses a deck of 8 cards. A coin flip determines who goes first. At the start of the match, each player can freely position their troops within 10 designated tiles. Players choose the turn order of their units (e.g., tank ‚Üí DPS ‚Üí healer), and this order remains fixed throughout the battle. The player always has 3 cards in hand. When there are no more cards in the draw pile, the discard pile is shuffled and becomes the new draw pile. Players can perform their actions in any order they wish: moving (limited by a number of tiles) and casting spells."

My Work on the Project

This game was developed as part of my master's degree in game development. We had two weeks to produce a prototype game in a team, based on a concept involving NFTs. From the beginning, we landed on the idea of creating a universe centered around cats, with two main phases: a village that serves as the HUB, and a combat phase in an arena. The goal was to use NFTs as unique representations of each cat, which could be healed, customized, traded, or used in battles.

Unity was already familiar to me, but this project allowed me to grow technically. I was mainly responsible for the following aspects:

üêæ Entity Management (Cats and Spells)

I implemented a flexible system to manage a variable number of cats and spells. Each cat had stats, affinities with certain buildings and spells, and its own visual representation. These entities had to be dynamically instantiated in the UI, which pushed me to build a modular and scalable interface.

üñ•Ô∏è User Interface (UI)

I moved from working with static interfaces to creating a responsive and adaptive UI, capable of reacting to an increasing number of cats and spells, and to real-time game events. Key UI elements included:

  • Dynamic lists based on saved data
  • Smart UI refresh system (updates on data changes without performance loss)
  • Visual feedback for interactions (drag & drop, hover tooltips, transitions)

üíæ Data Persistence

This was my first time implementing a save/load system between game sessions. I used a local JSON-based serialization system to:

  • Save stats, spells, upgraded buildings, and cat assignments
  • Restore the village state and UI on game launch
  • Track player progression like unlocked cats and building states

‚öôÔ∏è Background Logic

Village buildings needed to continue producing resources while the player was away. I integrated a system based on the system clock to calculate elapsed time and generate resources accordingly when the game restarts.

Conclusion

This project was highly educational. In just two weeks, I was able to:

  • Deepen my skills in dynamic and responsive UI development
  • Learn how to handle persistent data and multi-system interactions
  • Collaborate effectively with a team on a technically and creatively ambitious project

It also gave me a clearer understanding of the challenges related to progression systems and persistent gameplay, similar to mobile and management games.
